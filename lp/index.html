伝票作成・印刷・PDF取得・12桁追跡番号の取得・削除まで、 ブラウザ UI 相当の操作を REST / MCP でカバーします。

GitHub で見る →
✅ 実機検証済み（E2E 約 20秒完走） ✅ Node.js 20 / TypeScript strict ✅ Apache 2.0

特長

ブラウザ UI と完全互換のプロトコル、実機で踏み抜いた落とし穴を全部回避済み。

🔐

CSRF ヘッダ自動付与

Origin / Referer / X-Requested-With を自動付与。欠けると 417 Expectation Failed になる罠を回避。

📦

JSON + msgpack+zlib

通常は JSON（$.b2fetch 互換）。大量一括 / DELETE は msgpack+zlib（MPUploader 互換）を自動切替。

🖨

プリンタ種別自動切替

output_format: "label" 指定で general_settings.printer_type を切替→印刷→元に戻す。

🔁

自動再ログイン

Cookie 失効 (401/403) を検知して自動的に再ログイン → リトライ。セッション管理不要。

📄

PDF 2段階取得

checkonly=1 → fileonly=1 の 2段階を自動化。新規印刷でも checkonly 無しだと 96B HTML エラー。

🎯

12桁追跡番号取得

polling Success では内部番号のまま。PDF 取得後 1.4〜2.6秒で 12桁が割当される正しいトリガーを実装。

⚡

Vercel ワンボタン

環境変数 3 つ（B2 認証 + MCP APIキー）入れるだけで即稼働。Serverless Functions で運用コストほぼゼロ。

🤖

MCP サーバー搭載

Claude Desktop / Cursor / Continue 等から tools/call で即利用可能。ツール 12 個を JSON-RPC で提供。

✅

型安全 + テスト

TypeScript strict、Zod 入力検証、vitest 65 ケース。ネットワーク非依存の単体テスト完備。

Quick Start

3 分で立ち上がり、即座に伝票作成できます。

① Deploy with Vercel

上のボタンから、環境変数 B2_CUSTOMER_CODE / B2_CUSTOMER_PASSWORD / MCP_API_KEY の 3 つを入力するだけ。

② ローカル開発

git clone https://github.com/DaisukeHori/b2cloud-api.git
cd b2cloud-api
npm install
cp .env.example .env
# .env を編集
npm run dev

③ 伝票を作成して 12桁追跡番号を取得

curl -X POST https://your-app.vercel.app/api/b2/print \
  -H "Content-Type: application/json" \
  -d '{
    "shipments": [{
      "service_type": "0",
      "consignee_name": "テスト太郎",
      "consignee_zip_code": "100-0001",
      "consignee_address1": "東京都",
      "consignee_address2": "千代田区",
      "consignee_address3": "千代田1-1",
      "item_name1": "サンプル商品"
    }],
    "print_type": "m5"
  }'

# → { "results": [{
#     "tracking_number": "389711074012",
#     "issue_no": "UMIN...",
#     "pdf_base64": "JVBERi0x..."
#   }] }

認証

API キーと B2 ログイン情報の仕組み。

API キー (MCP_API_KEY)

条件API キー
env var に B2_CUSTOMER_CODE あり必須 — クエリ ?key=xxx またはヘッダー X-MCP-API-Key
env var に B2_CUSTOMER_CODE なし不要 — 守る情報がないので公開OK
env var に MCP_API_KEY なし不要 — 認証自体が無効

env var に B2 ログイン情報を入れずにデプロイすれば、API キーなしの公開 API になります。
その場合、呼び出し側がヘッダーで B2 ログイン情報を毎回渡します。

B2 ログイン情報の渡し方

お客様コード・パスワードは3箇所のどこかに入っていれば動く。優先順位:

優先度場所ヘッダー / 変数名
1（最優先）リクエストヘッダーX-B2-Customer-Code / X-B2-Customer-Password
2Vercel 環境変数B2_CUSTOMER_CODE / B2_CUSTOMER_PASSWORD
—どちらもなしエラー

環境変数にデフォルト値を入れておけばリクエストごとに送る必要なし。入っていてもヘッダーで上書き可能。

送り状に必要な項目

宛先 + 品名の 8 フィールドだけで送り状が出る。依頼主・請求先は環境変数でデフォルト設定済。

REST API

12 個のエンドポイントで B2クラウドのすべての主要操作をカバー。

MethodPath用途
GET/api/healthヘルスチェック
POST/api/b2/loginログインしてセッション確立
POST/api/b2/checkcheckonly（バリデーションのみ）
POST/api/b2/savecheck → 保存
POST/api/b2/printcheck → 保存 → 印刷 → PDF → 12桁追跡番号（フル E2E）

GET / PUT/api/b2/settingsプリンタ設定取得 / 切替

実機で踏み抜いた落とし穴

Node.js E2E 検証で実際に遭遇して、本ライブラリが回避済みの罠。

  1. CSRF ヘッダ必須 — Origin / Referer / X-Requested-With が無いと 417 Expectation Failed
  2. 認証は5段階 — bmypage GET → bmypageapi POST → 302追跡(5ホップ) → ME0002(form-urlencoded) → serviceUrl(OAuth) → template.dat
  3. テンプレート URL — /tmp/template.dat (base64, 460行) が正。/b2/d/_settings/template (1115行) とは別物
  4. author{} の子要素はスペース付き — " name"/" uri"/" email"、スペース無しで entry idx が 3 個ズレる
  5. 302 リダイレクト — redirect: 'manual' で手動追跡しないと Cookie が途中で失われる
  6. PDF は 2段階取得 — checkonly=1 → fileonly=1。新規印刷でも checkonly 無しだと 96B HTML
  7. 12桁追跡番号は PDF 取得後 — polling Success は UMN 内部番号、PDF 取得 1.4〜2.6 秒後に反映
  8. search_key4 は 16文字以内・英数字のみ — 17文字や記号で ES002070
  9. DELETE は msgpack+zlib 必須 — JSON body では 409 or 実削除されない
  10. invoice_code_ext は空文字 — 枝番は invoice_freight_no 側に入れる
  11. タイム便の delivery_time_zone — "0010" / "0017" のみ
  12. polling の feed.title="Error" は正常 — summary="queued" は処理中、throwOnFeedError: false でリトライ必須
